# テスト観点チェックリスト生成スキル

リポジトリのfork/clone元からの全変更差分を分析し、テスト観点を網羅的に洗い出す。
観点の種類ごとに個別のCSVファイルを生成する。

## 手順

### 1. 変更差分の収集

fork/clone元からの全変更を把握する。以下の優先順で差分の起点を決定する:

```bash
# 1. fork元のリモートがあればそこから
git remote -v

# 2. upstreamがあればupstream/mainとの差分
git merge-base HEAD upstream/main 2>/dev/null

# 3. origin/mainとの差分（最も一般的）
git merge-base HEAD origin/main 2>/dev/null

# 4. どれもなければ最初のコミットから
git rev-list --max-parents=0 HEAD
```

起点が決まったら差分を取得する:

```bash
git log <起点>..HEAD --oneline
git diff <起点>...HEAD --stat
```

変更ファイルが多い場合は、Exploreエージェントを使って主要な変更ファイルの中身を並行で読み取り、各機能・修正の詳細を把握する。

### 2. 変更内容の分類

変更内容を以下の軸で分類する:

- **新機能**: 新規追加されたファイル・コンポーネント・API
- **修正**: 既存機能の不具合修正
- **リファクタリング**: 動作は変わらないがコード構造の変更
- **UI/UX**: 画面表示・操作性の変更
- **データ**: シードデータ・定数・設定値の変更
- **通知**: メール・プッシュ通知テンプレートの変更
- **認証・セキュリティ**: 認証フロー・アクセス制御の変更
- **インフラ・設定**: 環境変数・CI/CD・Docker設定の変更

### 3. CSV出力（観点の種類ごとにファイルを分割）

以下のCSVファイルをすべて生成する。プロジェクトに該当しないカテゴリのファイルは省略してよい。

---

#### ファイル1: `test-checklist-flow.csv` — 基本フロー

アプリケーションの主要な操作フローを網羅する。
「AしてBしてCになること」のように、一連の流れを1行で表現する。

含めるべきパターン:
- ハッピーパス（最も基本的な操作の一連フロー）
- 主要な分岐を含むフロー
- CRUD操作の組み合わせ（作成→更新→完了 / 作成→削除 / 作成→削除→再作成→完了 など）
- 複数機能を横断するフロー
- プロジェクト固有のドメインフロー（変更差分から読み取る）

---

#### ファイル2: `test-checklist-boundary.csv` — 境界値・条件分岐

「この場合はこうなること」の形式で、境界値や条件の違いによる挙動を列挙する。

含めるべきパターン:
- 期限・時刻系: 期限1秒前に操作→成功すること / 期限直後に操作→不可であること / 期限ちょうどに操作→どちらになるか
- 数量系: 残り1件→操作成功→0件になること / 0件→操作不可であること / 上限値ちょうど→受け付けること / 上限+1→拒否されること
- 入力値系: 空文字→バリデーションエラーになること / 最大文字数→受け付けること / 最大文字数+1→エラーになること / 特殊文字(絵文字・記号)→正しく処理されること
- 状態系: 初回操作→成功すること / 2回目の同一操作→適切にハンドリングされること / データ0件時→空状態が表示されること
- 日付系: 過去日付→エラーになること / 当日→受け付けること / 未来日付→受け付けること / タイムゾーン境界での挙動

---

#### ファイル3: `test-checklist-auth.csv` — 認証・権限・アクセス制御

認証状態や権限の違いによる挙動を列挙する。変更差分に認証・権限関連の変更がない場合は省略可。

含めるべきパターン:
- 未認証→保護ページアクセス→ログインページにリダイレクトされること
- 認証済み→保護ページアクセス→正常に表示されること
- セッション切れ→操作→再認証を求められること
- 再認証後→元の操作に戻れること
- 権限なしユーザー→操作→拒否されること
- 認証トークン期限切れ→自動リフレッシュされること（該当する場合）
- ログイン→ログアウト→ブラウザバック→保護ページが表示されないこと
- 複数タブでのログイン/ログアウトの整合性

---

#### ファイル4: `test-checklist-error.csv` — 異常系・エッジケース

異常な操作やエラー時の挙動を列挙する。

含めるべきパターン:
- 二重送信: フォーム送信ボタン連打→1件のみ登録されること
- ブラウザバック: 完了画面からブラウザバック→再送信されないこと
- 不正アクセス: 存在しないIDでアクセス→適切なエラー画面が表示されること
- 不正パラメータ: URLパラメータ改ざん→エラーハンドリングされること
- ネットワーク: 送信中にネットワーク断→エラー表示されること / 再接続後にリトライ可能であること
- 同時操作: 2ユーザーが同時に同一リソースを操作→データ不整合にならないこと
- 大量データ: 一覧画面に大量データ→パフォーマンスが劣化しないこと / ページネーションが正しく動作すること
- 初期化失敗: 外部SDK初期化失敗→適切なエラー画面が表示されること

---

#### ファイル5: `test-checklist-notification.csv` — 通知・メッセージ

通知やメッセージ送信に関する観点。変更差分に通知関連の変更がない場合は省略可。

含めるべきパターン:
- 各操作後に正しい通知が届くこと（内容・宛先・タイミング）
- 通知内の動的データ（名前・日時・金額等）が正しく埋め込まれていること
- 特殊文字を含むデータが通知内で文字化け・エスケープ崩れしないこと
- 操作の組み合わせで通知が重複・混同しないこと（例: 更新後にキャンセル→キャンセル通知のみ届くこと）
- 通知チャネルごとの表示確認（メール・プッシュ・アプリ内通知等）

---

#### ファイル6: `test-checklist-display.csv` — 表示・UI

画面表示に関する観点。UI変更がある場合に生成する。

含めるべきパターン:
- 各画面で正しい情報が表示されること
- データが0件の場合の空状態表示
- ローディング中の表示
- エラー時の表示
- モーダル・ダイアログの開閉が正しく動作すること
- モーダル表示中に背面の操作がブロックされること
- スクロール（モバイル含む）が正しく動作すること
- 動的に切り替わる表示（有効/無効、表示/非表示の条件分岐）

---

#### ファイル7: `test-checklist-critical.csv` — 重要度: 高（必須確認）

上記すべてのCSVから、特に重要度が高いものを抜粋してまとめる。
以下の基準で選定する:

- **データ損失リスク**: 二重登録、データ消失、不整合
- **金銭的影響**: 課金、決済、ポイント計算（該当する場合）
- **セキュリティ**: 未認証アクセス、権限エスカレーション
- **ユーザー影響大**: メインフローが動かない、操作不能になる
- **復旧困難**: 一度発生すると元に戻せない操作

このファイルは他のCSVと重複してよい。「最低限これだけは確認する」リストとして使う。

---

### 4. CSVフォーマット（全ファイル共通）

```
確認済,観点
,カテゴリ名
,観点の説明文
,観点の説明文
,
,次のカテゴリ名
,観点の説明文
```

- **確認済列**: デフォルトは「確認済」の1列。ユーザーが `$ARGUMENTS` で列のカスタマイズを指示した場合はそれに従う（例: "確認済列を3環境にして" → `dev,stg,prod,観点`）
- **カテゴリ行**: 確認済列は空、観点列にカテゴリ名を記載（見出し行）
- **観点行**: 確認済列は空、観点列に説明文をそのまま記載（先頭に `- ` を付けない）
- **空行**: カテゴリ間に空行を1行入れて視認性を確保

### 5. 出力後の案内

CSVファイル作成後、以下を案内する:
- 生成したファイル一覧（ファイル名・項目数・カテゴリ数）
- 省略したファイルがあればその理由
- 「スプレッドシートにインポートして確認済列にチェックボックスを設定してください」の案内

## 引数（$ARGUMENTS）の解釈

引数は自由文で追加指示を受け付ける。以下は代表的な指示例:

- `"確認済列を4会場(銀座,大阪,京都,原宿)にして"` → 確認済列を4列に変更
- `"認証周りを重点的に"` → 認証関連の観点を厚くする
- `"通知は不要"` → 通知関連のファイルを省略
- `"E2Eで自動化する前提で"` → 自動テスト向けの粒度で記載
- 引数なし → デフォルト設定（確認済1列、全ファイル生成）で実行

## 注意事項

- テスト手順（操作ステップ）は不要。観点（何を確認するか）のみ記載する
- 境界値の観点は「〜の場合→〜になること」の形式で具体的に書く
- 1つの観点は1行で完結させる。長すぎる場合は分割する
- プロジェクト固有の用語はコードから読み取ってそのまま使用する
- 日本語で記載する
- 該当しないカテゴリのファイルは生成しない（ファイル数を無駄に増やさない）
- `test-checklist-critical.csv` は他ファイルとの重複OK。最低限の確認リストとして独立させる
